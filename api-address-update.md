# API 地址改进记录

## 本次已完成

- 已把前端 API 地址从多个服务文件中的硬编码，收口到统一配置模块：
  - `client-app/config/api.js`
- 已替换以下调用方，改为统一读取 `API_BASE_URL`：
  - `client-app/services/auth/api.js`
  - `client-app/services/binding/api.js`
  - `client-app/services/message/api.js`

## 当前实现方式

- 默认 API 地址仍为：`http://127.0.0.1:3000`
- 前端不再在多个文件里重复写这个地址
- 当前统一读取规则为：
  1. 先读取运行时配置 `globalThis.__CHAT_APP_CONFIG__.apiBaseUrl`
  2. 若未提供，再回退到默认值 `http://127.0.0.1:3000`

这意味着后续如果需要切换环境，只需要改一个配置入口，不需要逐个服务文件修改。

## 本次改进解决了什么

- 消除了多处硬编码地址
- 降低了改端口、改主机时的漏改风险
- 为后续接入多环境配置留出了清晰入口

## 当前仍然存在的限制

- 默认值仍然是本地开发地址 `http://127.0.0.1:3000`
- 目前项目还没有正式的环境变量注入机制
- 目前也还没有使用相对路径和开发代理

## 后续建议方案

### 方案一：相对路径 + 开发代理

推荐优先级最高，适合后续继续演进：

- 前端改为请求相对路径，例如 `/api/messages/poll`
- 前端开发服务器把 `/api` 代理到后端服务
- 生产环境由反向代理或网关统一转发

优点：

- 前端不再关心主机和端口
- 本地开发、测试环境、正式环境切换更自然
- 更接近正式部署架构

### 方案二：运行时注入配置

适合需要一份前端产物部署到多个环境时使用：

- 在 `index.html` 或服务端模板中注入：
  - `globalThis.__CHAT_APP_CONFIG__ = { apiBaseUrl: "..." }`
- 前端启动时读取该配置

优点：

- 不需要为不同环境重复改源码
- 可以让同一份前端静态资源适配不同后端地址

### 方案三：服务端同源托管

适合更完整的部署收敛方案：

- 让后端统一托管前端资源
- 前端接口全部走同源路径
- 减少 CORS 和跨端口调试成本

## 建议的下一步

如果后面继续推进，我建议顺序是：

1. 保持本次统一配置模块不变
2. 下一步把接口逐步改成相对路径
3. 给 `client-app/dev-server.mjs` 增加 `/api` 代理
4. 最后再按部署需求决定是否接入运行时配置注入
